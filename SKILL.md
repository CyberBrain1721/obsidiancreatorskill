# 文境（Wenjing）

文境是一套协议驱动的 AI 协作创作操作系统。协议决定流程，Agent 负责执行协议——不主动决定、不自行推断、不扫描全库。

## 核心理念

**第一个念头，永远是人的。**

AI 帮你走得更远。但它不替你走出第一步。

- 收录，是你做的第一个判断（「这个东西让我有兴趣」）
- 共创，是你先看见的连接（AI 不先于你提炼方向）
- 成卷，标题和结构是你确认的（AI 不跳过确认）
- 入典，是你验证过的东西才入库
- 整理，删除权在你手里

## 触发条件

当用户说以下任一内容时加载本 Skill：
- 「文境入口」「进入文境」「文境」
- 「启动文境」「创建文境」「初始化文境」
- 「收录」「共创」「成卷」「入典」「整理」
- 「刷新看板」

## 一、协议初始化（Vault 已存在时）

当 Vault 已存在（非首次启动）时，执行协议初始化流程。

### 阶段 1：入口

核心原则：
- 不主动索引整个 Vault
- 不根据已有知识推断当前任务
- 不自行决定下一步工作流程
- 协议驱动（Protocol-Driven）：Agent 执行协议

### 阶段 2：模式选择

呈现两个选项：
- ① 共创模式：作品创作、灵感讨论、知识整理、项目推进
- ② 开发模式：系统开发、协议修改、架构评审、Bug修复

**等待用户选择，不继续读取。**

### 阶段 3A：共创模式

进入 Creator Runtime：

1. 读取 Workflow Index（`04 Agent入口/05 Workflow Index.md`），了解五类 Workflow 触发条件
2. 进入 Conversation Runtime（默认状态），持续监听创作者任务
3. Lazy Loading：所有 Workflow 仅按需加载、按需执行，完成后立即返回 Conversation Runtime

初始化完成后进入 **Ready for Collaboration** 状态。

## 二、Bootstrap：首次启动

当用户在新设备上首次说「启动文境」时，执行以下流程：

### 第 0 步：检查 Obsidian

确认用户设备上已安装 Obsidian。如未安装，直接给出下载链接：

```
⚠ 未检测到 Obsidian。请先下载安装：https://obsidian.md/download
安装完成后创建一个新的 Vault，然后回来说「启动文境」。
```

### 第 1 步：确认路径

```
请指定文境 Vault 的存放路径（留空则在 Documents 下自动创建 `WenjingVault`）：
```

- 用户指定路径 → 使用该路径
- 用户留空 → 在 `~/Documents/WenjingVault`（Windows: `C:\Users\<user>\Documents\WenjingVault`）自动创建
- 路径已存在且含 `.obsidian/` → **更新模式**：跳过目录创建，仅同步最新模板文件

### 第 2 步：一键 Bootstrap

运行 `scripts/bootstrap.py`，一次性完成目录创建、种子文档写入、插件安装、Obsidian 配置：

```bash
python scripts/bootstrap.py "VAULT_PATH"
```

输出示例：
```
Vault: C:\Users\...\WenjingVault

  ✓ 目录结构已创建
  ✓ 16 个种子文档已写入
  ✓ 7 个插件已安装
  ✓ Obsidian 配置已写入

文境已就绪。重启 Obsidian 后打开 文境看板.html。
```

### 第 3 步：设置看板刷新

Cron job 默认每天一次运行 `scripts/refresh_dashboard.py`。

### 第 4 步：进入协议初始化

Bootstrap 完成后，执行「协议初始化」流程（阶段 1-2），进入共创模式。

## 三、五个指令

### 1. 收录

**触发**：「收录 [内容]」「保存到藏阁」「记录」

**Agent 执行**：直接用 `write_file` 写入 Inbox，不需要脚本。
1. 文件路径：`02 养境（知识）/01 藏阁/01 Inbox/【类型】标题.md`
2. 类型从：念/摘/事/梦/感/随/文/对/白/图 中选最匹配的
3. YAML frontmatter（Obsidian Properties 扁平键值对，禁止嵌套）：

```yaml
---
status: 待发展 | 共创中 | 成卷 | 已归档
type: 念 | 摘 | 事 | 梦 | 感 | 随 | 文 | 对 | 白 | 图
resource_url:        ← 仅当有真实 URL 时保留，空值则删除整行
resource_capture_time:
resource_policy:
tags:
publish_date:
confidence:
---
```

禁止字段：`resource_type`、`language`、`resource_id`、`resource_source`、`canonical`、`related`、`author`。

4. 正文结构：`# 【类型】 标题` → `## Resource` → `## Original Content` → `## Summary` → `## Insight` → `## References`

**Agent 不做什么**：
- 不追问「为什么收」——收录本身就是判断
- 不替创作者评估价值
- References 无关联时留空，不填入 `[[AI]]` 等 KM 节点

### 2. 共创

**触发**：「共创」

**执行原则：AI 不先于创作者完成第一次模式识别。**

1. 扫描 Inbox 中 `status ≠ 成卷` 的条目，随机抽 3 条
2. **只展示原文摘要**（标题 + 关键段落），不提炼、不生成方向
3. 问：「你看到了什么？它们之间有什么联系？」
4. 等创作者表达直觉后，再展开/补充/连接

**正确示例**：
```
从藏阁随机抽取三条：

①【摘】闻道有先后
   > "信息本身并不能改变行为。知道和做到之间的鸿沟……"

②【念】无限答案时代的认知构建
   > "当 AI 可以瞬间生成任何答案，真正稀缺的不再是获取答案的能力……"

③【摘】暗处生长
   > "所有真正重要的变化，都不是在聚光灯下完成的……"

你看到了什么？它们之间有什么联系？
```

**错误（AI 越界——替创作者走了第一步）**：
```
三个方向：
① 知识的应用鸿沟
② 认知构建 vs 消费
③ 暗处生长的力量
```

### 3. 成卷

**触发**：「成卷」

1. 生成 3 个标题方向 → 等用户选择
2. 生成章节结构 → 再等确认
3. 确认后一次性生成正文
4. 创建卷目录：`03 成卷（项目）/卷NNN · 标题简写/`
5. 创建两个文件：`创作过程：标题.md`（YAML+共创记录，含 `aliases` 防 wikilink 断裂）+ `正文：标题.md`
6. 来源 Inbox `status` → `成卷`，References 双向链接，保留未选方向

**两步确认缺一不可。**

### 4. 入典

**触发**：「入典 [描述]」

**核心判断**：一年后，我还需要引用它吗？

三条路径：

| 路径 | 来源 | 示例 |
|------|------|------|
| 从卷提炼 | 卷完成后提取可复用方法 | `入典 从卷003提炼 AI学习四门课框架` |
| 直接沉淀 | 客观规则/法律条文 | `入典 《著作权法》第22条` |
| 主动入库 | 自己总结的方法论 | `入典 React useEffect 清理模式` |

典阁分类：创作体系 / 技术知识 / 法律与规则 / 客观知识 → 写入 `02 养境（知识）/02 典阁/01 Library/[分类]/`

**典阁不是第二个藏阁**。未经验证的东西不入典。

### 5. 整理

**触发**：「整理」「分类」「重构」

扫描 Inbox，按 type/tags 分类展示，建议归并和状态更新。**删除权永远在创作者手里。**

## 四、文境看板

两个动态仪表盘：`文境看板.html`（全貌）+ `文境速览.html`（精简）。

- **手动刷新**：说「刷新看板」→ 运行 `scripts/refresh_dashboard.py`
- **自动刷新**：Bootstrap 时设置的 Cron job，默认每天一次
- **重要**：看板用**浏览器**打开，非 Obsidian HTML 插件。所有 Obsidian HTML 渲染插件都运行在沙箱 iframe 中，`obsidian://` 链接被阻断——这是安全设计，无法绕过。

## 五、系统约束

**AI 不替创作者走第一步。**

- 决策权：AI 给建议，人决定
- 收敛约束：共创阶段不主动生成完整成品
- 阶段控制：AI 不主动推进流程
- 边界约束：不主动总结「你的整体方向」
- 缓判原则：AI 一次建议 ≠ 创作方向，留讨论空间
- 多任务冲突：先执行低阶行为（记录），暂停高阶行为（生成），向创作者确认

## 六、常见陷阱

1. 模式选择前索引文档 → 必须等待用户选择
2. 共创时 AI 替创作者提炼方向 → 展示原文摘要，先问「你看到了什么」
3. Capture 输出跳过 Inbox Standard → 必须先读格式规范
4. Inbox YAML 使用嵌套结构 → Obsidian Properties 只识别扁平键值对
5. References 填入 KM 节点填充 → 无关联时留空
6. Vault 根目录新增文件 → 绝对禁止，除看板外
7. 批量修改同类文件前未询问 → 必须先确认
8. 成卷后 Inbox ↔ 卷双向链接断裂 → status 更新 + References 双向链接
9. `sed -i` 在 Windows/MSYS 损坏 UTF-8 文件 → 用 `patch` 或 Python `open()`
10. `execute_code` 内 `read_file` 返回行号前缀 → 用 Python 原生 `open()`
11. 卷文件重命名导致 wikilink 断裂 → 创作过程文件 YAML 必须声明 `aliases`
12. 典阁变成第二个藏阁 → 只收已验证、可长期复用的知识
13. 试图在 Obsidian HTML 插件中让看板链接可点击 → 不可能，用浏览器打开看板
14. 试图用 CSS Snippet 复刻看板布局 → Obsidian 默认样式冲突严重，不要走这条路
15. 同一任务失败 3 次必须换方案 → 不反复修补同一个思路

## 八、核心原则

1. **第一个念头，永远是人的**：AI 不替创作者走出第一步
2. **协议驱动**：Agent 执行协议，不主动决定流程
3. **不主动决定流程**：协议决定流程，Agent 负责执行

5. **规范延续**：不重新设计已确认规范
6. **一致性优先**：不因回答当前问题破坏既有系统一致性
